# React-redux源码阅读

Redux 是很常用的 ReactJs 状态管理库，在实际使用的时候通常结合 React-redux 一起使用。今天开始拜读一下 React-redux 的源码。

# Provider

在使用的时候，最先就是要用 <Provider/> 组件包裹起来，传入 Redux 生成的 store。

Provider 的主要作用，就是利用 React 的 context 给之后 connect 组件的使用提供上下文，看看，Provider 究竟给它的子孙后代提供了什么东西：

function Provider({ store, context, children }) {
  const contextValue = useMemo(() => {
    const subscription = new Subscription(store)
    // redux 返回的store中有一个 subscribe 方法用于监听 store 的改变 每一次dispatch都会触发这个方法
    subscription.onStateChange = subscription.notifyNestedSubs
    // 这一步做到了 store  subscribe 方法出发的时候 出发 notifyNestedSubs 方法 出发所有嵌套的监听
    return {
      store,
      subscription
    }
  }, [store])
  //....
  const previousState = useMemo(() => store.getState(), [store])

  useEffect(() => {
    const { subscription } = contextValue
    subscription.trySubscribe()

    if (previousState !== store.getState()) {
      subscription.notifyNestedSubs()
    }
    return () => {
      subscription.tryUnsubscribe()
      subscription.onStateChange = null
    }
  }, [contextValue, previousState])
  //....
}

可以看到，context保存的是 store对象，以及 Subscription 的实例。其实看 Provider 的源码很明显的可以发现了，就是生成了一个 subscription 然后给 Redux 的 store 绑定了 Subscription notifyNestedSubs 方法，看到这里其实就可以判定，后续的 connect 的监听都是绑定在 Provider 创建的这个 Subscription 实例的监听队列中的。

# ConnectAdvanced

说实话，看源码之前我并不知道有这个 Api。也是这次看源码才知道平时我们熟知的 connect 方法其实这是基于这个方法传入一组特定的参数去实现的。看这个方法的源码之前，先看看这个方法之前，先来到官网看看这个方法大概有哪些参数，具体如何使用。

import * as actionCreators from './actionCreators'
import { bindActionCreators } from 'redux'

function selectorFactory(dispatch) {
  let ownProps = {}
  let result = {}

  const actions = bindActionCreators(actionCreators, dispatch)
  const addTodo = text => actions.addTodo(ownProps.userId, text)

  return (nextState, nextOwnProps) => {
    const todos = nextState.todos[nextOwnProps.userId]
    const nextResult = { ...nextOwnProps, todos, addTodo }
    ownProps = nextOwnProps
    if (!shallowEqual(result, nextResult)) result = nextResult
    return result
  }
}
export default connectAdvanced(selectorFactory)(TodoApp)

这是官方文档上一段对于这个api的使用的示范例子。可以看到，这个方法需要我们自己传入一个方法，接受的应该是 redux 的 dispatch ，然后返回一个方法，又接收 state(应该是 Redux 的 Store 对象getState后得到的state)、还有一个 nextOwnProps 就暂且的不关心它。这个方法所返回的值应该就是传入被包裹组件中的props。可以看到，其中做了对于更新前后的值做了 shallowEqual 浅比较，后续会介绍这个方法。根据这里其实可以判断了 connect 是基于 ConnectAdvanced 的一个实现，其中为我们做了基础的更新前后比较（目的是为了不必要的更新，如果不做这些优化，每一次 dispatch 操作所有被 connect 的组件都会执行更新，即使 dispatch 改变的并不是它所依赖的那部分数据）。当我们想做一些更符合自己项目的优化操作的时候，就可以基于 ConnectAdvanced 去重新封装一个 connect 函数，在其中做更加贴合项目情况的优化操作。
