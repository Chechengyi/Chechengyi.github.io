# 对于 Webpack 打包文件的探究

Webpack 是在开发中经常使用到的工具，我们在开发中也经常会使用到模块化开发这种方式。那么 Webpack 打包产生的代码究竟是如何组织逻辑去实现 “模块化” 的呢？ 对于现在的不同模块化方案：如 AMD、requireJs、CMD 打包又有何不同呢？ UMD 模式下又是如何运作的？

带着这些问题，我决定开始分析一下不同场景下 Webpack 打包所产出的代码。

首先我写了一个很简单的 JS 文件，内容就是 console.log(1) ，就这么一行代码，Webpack 最终打包所产生的 JS 文件代码 为94行。

(function(modules){
  // runtime
})([
  function(){console.log(1)}
])

这就是打包所产生的代码的大体的样子，可以看到我们所写的代码是被打包带一个数组当中，而这个数组作为立即执行函数的参数modules传入到了函数当中。而 runtime 部分的内容，就是 Webpack 对于模块化实现的关键。

/******/ 	// The module cache 用来存储模块的对象
/******/ 	var installedModules = {};
/******/
/******/ 	// The require function 加载函数
/******/ 	function __webpack_require__(moduleId) {
/******/
/******/ 		// Check if module is in cache
/******/ 		if(installedModules[moduleId]) {
/******/ 			return installedModules[moduleId].exports;
/******/ 		}
/******/ 		// Create a new module (and put it into the cache)
/******/ 		var module = installedModules[moduleId] = {
/******/ 			i: moduleId,
/******/ 			l: false,
/******/ 			exports: {}
/******/ 		};
/******/
/******/ 		// Execute the module function
/******/    // 从这里可以看出 加载某一个模块世纪上是执行这个模块，设置了this指向 传入了 module 加载函数
/******/ 		modules[moduleId].call(module.exports, module, module.exports, __webpack_require__);
/******/
/******/ 		// Flag the module as loaded
/******/ 		module.l = true;
/******/
/******/ 		// Return the exports of the module
/******/ 		return module.exports;
/******/ 	}
/******/
/******/
/******/ 	// expose the modules object (__webpack_modules__)
/******/ 	__webpack_require__.m = modules;
/******/
/******/ 	// expose the module cache
/******/ 	__webpack_require__.c = installedModules;
/******/
/******/ 	// define getter function for harmony exports
/******/ 	__webpack_require__.d = function(exports, name, getter) {
/******/ 		if(!__webpack_require__.o(exports, name)) {
/******/ 			Object.defineProperty(exports, name, { enumerable: true, get: getter });
/******/ 		}
/******/ 	};
/******/
/******/ 	// define __esModule on exports
/******/ 	__webpack_require__.r = function(exports) {
/******/ 		if(typeof Symbol !== 'undefined' && Symbol.toStringTag) {
/******/ 			Object.defineProperty(exports, Symbol.toStringTag, { value: 'Module' });
/******/ 		}
/******/ 		Object.defineProperty(exports, '__esModule', { value: true });
/******/ 	};
/******/
/******/ 	// Load entry module and return exports
/******/  // 自动加载模块里面的第0个模块， 也就是加载入口 
/******/ 	return __webpack_require__(__webpack_require__.s = 0);

这部分的代码基本上不会改变，是 Webpack为了实现模块化的胶水代码

当我们将 js 文件改为：

export const a = 'b'
export default function(){
  console.log(a)
}

此时文件打包的内容：

function(module, __webpack_exports__, __webpack_require__) {

"use strict";
__webpack_require__.r(__webpack_exports__);
/* harmony export (binding) */ __webpack_require__.d(__webpack_exports__, "a", function() { return a; });
var a = 'b';
/* harmony default export */ __webpack_exports__["default"] = (function () {
  console.log(a);
});

/***/ }

Webpack 针对要打包的 js 文件生成了一个 module 对象，可以看到，这里其实就是将属性 a 放到了 module.export 对象里面，而 export default 的内容则是 module.export.default
