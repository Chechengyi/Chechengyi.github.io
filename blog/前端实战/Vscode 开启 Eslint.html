# Vscode 开启 Eslint

Eslint 是现在的前端代码中比较常用的代码校验工具，其目的是为了强制开发人员写出统一风格的代码。我上 Eslint 这条船主要是因为当 React Hooks 推出之后，在使用 Hooks 的途中可能会由于书写上的疏忽导致陷入 “闭包陷阱”（这并不是本篇文章想要记录的重点，如果感兴趣可以看我专门的一片关于闭包陷阱的文章, 链接）。

为了限制开发人员在使用 Hooks 的时候，陷入“不可预知”的错误，React 官方为我们 Eslint 的插件，它的主要作用就是在开发人员书写代码的时候，检测开发人员是够在条件语句中使用了 Hooks。或者是开发人员开使用 Hooks 的时候是否没有正确的设置它的依赖数组。如下图所示：

# Eslint 的配置文件

{
  "parser": "@typescript-eslint/parser",
  "env": {
    "browser": true,
    "commonjs": true,
    "es6": true,
    "node": true
  },
  "extends": [],
  "settings": {
    "react": {
      "pragma": "React",
      "version": "detect"
    }
  },
  "parserOptions": {
    "ecmaFeatures": {
      "experimentalObjectRestSpread": true,
      "jsx": true
    },
    "sourceType": "module"
  },
  "plugins": [
    "@typescript-eslint",
    "react",
    "react-hooks"
  ],
  "rules": {
    "react-hooks/rules-of-hooks": "error",
    "react-hooks/exhaustive-deps": "error",
    "indent": [
      "error",
      2
    ],
    "linebreak-style": [
      "error",
      "unix"
    ],
    "quotes": [
      "error",
      "single"
    ]
  }
}

# 开启 Eslint

在项目中开启 Eslint 可以下载包 eslint-loader，然后将这个包放到 babel-loader 和 ts-loader 的前面:

{
        test: /(\.tsx|\.ts)$/,
        use: ['babel-loader', 'ts-loader', 'eslint-loader']
      },

这样，在 Webpack 编译项目的时候，如果有错误就会在控制台显示错误。

但是其实最好用的还是 将我们常用的编辑器开启 Eslint，这样在编写代码的时候，编辑器就可以友好的给我们提示红色下划波浪线～（将rules配置为error），我就不信有开发人员看到这个不难受。比如我常用的就是 Vscode，首先 Vscode 下载 Eslint 插件。

但是，如果真的有那么顺利，今天的我也不会写下这篇文章了。

我在实际去配置的过程中，遇到了一个情况就是，明明我的 Eslint 已经开启了，并且在另外一个demo中都能成功的提示错误，但是在我一个本地项目中，就是不会提示错误。经过漫长的寻找也是才找到问题所在，这里记录一下。我解决的办法就是：

{
  "eslint.workingDirectories": [
    "./ziwu-web"
  ]
}

ziwu-web 是我的项目的文件夹的名称。关于这一点具体的介绍：https://marketplace.visualstudio.com/items?itemName=dbaeumer.vscode-eslint。其实看了我也没看的太明白。我大概的理解就是，可能我的这个项目由于文件夹嵌套（具体的触发的规则我还是不清楚）导致 Eslint 服务没有正确的解析到配置文件。所以增加这个配置以帮助 Eslint 确定工作目录。但是配置这个的时候最好是配置在 Vscode 的 WorkSpace 中，而不是配置到 User 中，不然如果在 User 中去指定目录，就会导致所有的项目都尝试去解析到这个目录。
