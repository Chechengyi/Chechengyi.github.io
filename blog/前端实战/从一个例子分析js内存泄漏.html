# 从一个例子分析js内存泄漏

内存泄漏，是 Javascript 开发人员经常谈及的一个话题。那么到底什么样的场景会造成内存泄漏，又应该怎么去防治内存泄漏呢？

我对内存泄漏的理解是这样的：变量是所有语言所提供的最基本的元素，在 Javascript 中，对于一些引用类型的数据或者是形成了闭包的变量所引用的数据，是保存在堆内存中的。当一个变量我们已经不再使用了，确由于开发时的一些疏忽导致这块儿数据所占用的内存空间无法被清理时，这就形成了内存泄漏。

# 1、闭包与内存泄漏

经常听见讲说，闭包会引起内存的泄漏，这曾经也给我留下了不小的阴影。让我在使用了闭包后总是战战兢兢，觉得是不是我写的东西内存泄漏了。

为什么开发者们会得出闭包会引起内存泄漏这个结论呢？

function test(){
		class Ccy {
		  	constructor(){
		  		this.name = 'ccy'
		  	}
		}
  	let ccy = new Ccy()
		return {
		    name: ccy
		}
}
let test1 = test()
// test1 = null

在这段代码中，实例 ccy 原本只是函数 test 的一个局部变量，如果没有 return 的这段代码，那么它们在函数 test 执行完毕之后，应该就会被销毁。但是现在函数 test 所返回的对象被保存到了变量 test1 上，而这个对象的 name 属性又引用了 ccy 这个实例， 这就形成了闭包，使得即使是基本类型的值，也被放到了堆内存当中，由于 test 的引用，所以这块儿内存也不会被销毁。

使用 Chrome 的控制台可以观察到这个实例。但是其实，当我们增加一行代码，也就是将 test1 设为 null 之后再去观察，机会发现 ccy 这块儿内存也会被正确的清理掉了。

所以，其实，并不是只要使用了闭包，就一定会导致内存泄漏。在这个例子中，只要最后所 return 出去的对象也不再被任何变量所引用，那么被闭包了的变量也会被垃圾回收器给清理的。所以，不要再对闭包有那么多的顾虑了。要知道，不会在被使用的变量才会被称之为垃圾。而对于闭包很多的使用场景当中，比如 去抖和节流经常会使用闭包的方法去记录一些数据。这些数据对于我们而言并不是没用的，所以它继续存在是合理的。
